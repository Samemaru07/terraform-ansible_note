## 1. IaC とは
**IaC (Infrastructure as Code)** とは、サーバ、ネットワーク、ストレージといったインフラの構成を、UI上の手動操作ではなく、ソフトウェア開発と同様に **「コード」を用いて定義し、構築管理を自動化する手法**のこと。

| 比較項目 | 手動管理 (ClickOps)        | IaC (コードによる管理)          |
| :--: | ---------------------- | ----------------------- |
| 作業方法 | 管理画面をポチポチする            | 設定ファイル (コード) を書く        |
| 再現性  | 手順書を見ながら再構築 (ミスが起きやすい) | コードを実行するだけで同じ環境が即座に完成   |
| 履歴管理 | 誰がいつ何を変えたか把握しにくい       | Gitで変更履歴が全て残り、以前の状態に戻せる |
| スピード | サーバ代数が増えると時間がかかる       | 1台でも100台でも、コマンド1つで一括構築  |

## 2. IaCを実現する3つの柱
IaCを理解する上で欠かせない「きほん」の要素は以下の3点。

### 1. 宣言型 (Declarative) の記述
多くのIaCツール (Terraformなど) は、「どう作るか (手順)」ではなく、**「どうあるべきか (完成形)」** を記述する。
- 例: 「サーバを1台追加する手順」を書くのではなく、「サーバが1台ある状態」とコードに書く。ツール側が現在の状態とコードの差分を読み取り、必要な操作を自動で判断。

### 2. べき等性 (Idempotency)
「同じコードを何回実行しても、必ず同じ結果になる」という性質。
既に設定済みのサーバに対して同じコードを実行しても、二重に設定が追加されたりエラーになったりせず、「既に設定済みである」と判断して何もしない。これにより、環境の安定性が保たれる。

### 3. バージョン管理 (Git)
インフラをコード化することで、GitHubなどのツールで管理できるようになる。
- **レビュー**: 変更内容を他のエンジニアが事前に確認できる。
- **ロールバック**: 設定ミスがあっても、過去の正常なコードに戻すだけで復旧できる。

## 3. なぜ今、IaCが重要なのか
現代のインフラは、クラウド (AWS, Azure, GCPなど) や仮想化技術によって、物理的な配線なしで動的にリソースを作成できるようになった。
- **ヒューマンエラーの排除**: 「設定し忘れ」「パラメータの打ち間違い」といった人のミスを防ぐ。
- **ドキュメントの自動化**: コードそのものが最新の構成図 (ドキュメント) の役割を果たす為、「手順書が古い」という問題が解決される。
- **スケーラビリティ**: サービスが成長してサーバを増やす際、同じコードを使い回すだけで瞬時に拡張が可能。

## 4. プロジェクトでの役割分担
- **Terraform**: 外箱の箱 (VPSのインスタンス、ネットワーク、ファイアウォール) を確保する。
- **Ansible**: 確保された箱の中身 (OSの設定、ミドルウェアのインストール、ファイルの配置) を整える。
