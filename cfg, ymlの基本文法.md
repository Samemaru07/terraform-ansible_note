## 1. `ansible.cfg`の文法と主要なキー
このファイルは、**INI形式 (Windowsの設定ファイル等でよく見られる形式)** で記述される。
コマンドを打つ時に、いちいちオプションを打たなくて済むようにする為に存在する。

### 基本文法
```cfg
[section_name]
key = value
```
### 主要なキー
|         キー名          |      意味       |                  役割                  |
| :------------------: | :-----------: | :----------------------------------: |
|     `inventory`      | インベントリファイルのパス |      どの名簿 (`hosts.yml`)を使うか指定する      |
|    `remote_user`     |  接続先OSのユーザ名   |        SSHログイン時のデフォルトユーザを決める         |
|  `private_key_file`  |   SSH秘密鍵のパス   |           合鍵 (秘密鍵) の場所を教える           |
| `host_key_checking`  |   ホストキーの検証    | `False`にすると、再構築後の「別のサーバ扱い」による警告を無視する |
| `interpreter_python` |   Pythonのパス   |       接続先で使うPythonを明示して、警告を消す        |

## 2. YAML (`.yml`) の基本ルール
YAMLの核となるのは、**「スペースによる相対的な階層定義」** と **「特定の記号直後のスペース** 。
1. **インデント**
	- **相対的な深さ**: 子要素は、親要素よりも右側にインデントされている必要がある。
	- **タブの禁止**: インデントタブにタブ文字を使用することはできない。必ず半角スペースを使用する。
2. **マップ** (`key = value`): 
	- **コロンの後のスペース**: `:`の直後には必ず1つ以上の半角スペース もしくは改行が必要。
3. **リスト (配列)**: 
	- **ダッシュの後のスペース**: `-`の直後には必ず1つ以上の半角スペースが必要。

```yaml
- name: <ログに表示されるタイトル
      パラメータ: 値
```

### 3. Ansible特有の記述ルール
### 真偽値 (Booleans)
YAML自体は`true`/`false`を扱いますが、Ansibleのドキュメントや慣習では`yes`/`no`が多様される。他にも`on`/`off`。だが、モジュール引数 (aptモジュールのupdate_chacheなど)に渡す引数は、yesでも良いが、LSPが怒ってくるときがある。

### 変数のクォート
Ansibleで変数`{{ var }}`を使用する際、行が`{{`で始まる場合は、必ずダブルクォートで囲む必要がある。
- 理由: YAMLにおいて、`{`はハッシュ (インライン形式のマップ) の開始記号である為、クォートがないと文法エラーになる。
- OK: `name: "{{ user_name }}"`
- NG: `name: {{ user_name }}`

### 複数行の記述
長いコマンドなどを複数行で書きたい場合に使用。
- `|` (リテラルブロック): 改行をそのまま保持。
- `>` (折り返しブロック): 改行を半角スペースに置き換え、1行として扱う。

## 4. 文法チェックの方法。
- `ansible-lint <ファイル名>`: Ansibleのベストプラクティスと文法をチェック。
- `ansible-playbook <ファイル名> --syntax-check`: 実行せずにPlaybookの文法エラーのみを抽出。
