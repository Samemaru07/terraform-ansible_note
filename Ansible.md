## 3つの神器
- **Inventory (インベントリ)**
	操作対象のIPアドレスやユーザ名を書いた名簿。
		「どの建物 (サーバ)」に工事に行くかのリスト
- **Playbook (プレイブック)**
	「パッケージを入れる」「設定を置く」という手順書。
		「どんな内装 (設定)」にするかの指示書
- **Role (ロール)**
	プレイブックを機能ごとに分けた部品 (共通設定, Nginx用など)
		「キッチン担当」「電気工事担当」などの専門職チーム

## Ansibleのpingは普通のpingではない
OS標準の`ping`コマンドは、ICMPというプロトコルを使い、「生きてる？」と聞くだけ。
しかし、Ansibleの`ping`モジュールは以下のステップを踏んでいる。
1. **SSH接続**: 指定した秘密鍵を使ってログインする。
2. **Python探索**: 相手のOS (EC2) の中に、指示を解釈できるPythonが入っているか探す。
3. 小規模プログラムの実行: 短いPythonスクリプトを送り込んで実行し、その結果を回収する。
`pong`という返答は、「SSHでログインできて、Pythonも動くから、俺 (Ansible) はいつでも仕事を始められるぜ!」という準備完了の合図。

## Ansibleの心臓「べき等性 (Idempotency)」
`ansible all -m ping`で返ってくるコマンドには`"changed": false`というのがある。
#### 「動作」ではなく、「状態」を定義
一般的なシェルスクリプトとAnsibleの違いは、
- **シェルスクリプト (手続き型)**:
	- 「`apt install nginx`を実行せよ」
	- 既に入っている場合、エラーが出たり、無駄に再インストールが走ったりする。
- **Ansible (宣言型)**:
	- 「`nginx`が**インストールされている状態**にせよ」
	- Ansibleはまず現状を確認し、既に入っていれば「何もしない (`changed: false`)」、なければ「入れる(`changed: true)」。
	- `lineinfile`では、`regexp`で指定したパターン、またはlineで指定した文字列そのものがファイル内に存在するかを確認。

### Ansibleの動き (プッシュ型)
Ansibleには「エージェント」不要。
1. **Local**: YAML (Playbook) を書く。
2. **Transfer**: Ansibleがその指示をPythonコードに変換し、SSH経由でEC2の`/tmp`等に送り込む。
3. **Execute**: 送り込まれたコードがEC2上で実行される。
4. **Cleanup**: 実行が終わると、送り込んだコードは自動で削除される。

> この「跡形も残さない」スマートさが、Ansibleが多くの現場で採用されている理由。

## ファイル間の指示の流れ
| **ファイル名**         | **役割**                            | **日常生活に例えると**                          |
| ----------------- | --------------------------------- | -------------------------------------- |
| **`ansible.cfg`** | **Ansible自体の設定**（秘密鍵の場所、警告の非表示など） | **職人の道具箱の設定**（どの鍵を使うか、作業着はどうするか）       |
| **`hosts.yml`**   | **接続先の名簿**（IPアドレス、ホストのグループ分け）     | **連絡先一覧**（誰の家がどこにあるか、住所の一覧）            |
| **`site.yml`**    | **全体の設計図**（どのサーバーに、どの役割を与えるか）     | **工事の全体工程表**（Aさんの家には電気工事と水道工事を依頼する）    |
| **`main.yml`**    | **具体的な作業手順**（パッケージを入れる、設定を変える）    | **現場の手順書**（1.壁紙を剥がす、2.ノリを塗る、3.新しい紙を貼る） |
1. `ansible.cfg`が読み込まれ、`hosts.yml` (名簿) の場所が特定される
2. `site.yml`を指定して実行 (`ansible-playbook site.yml`)
3. `site.yml`が「allグループ にcommonロールを適用せよ」と命令
4. Ansibleが`roles/common/tasks/main.yml` (手順書) を読み込み、名簿に載っているサーバに対して、1行ずつプッシュを送り込む

## ハンドラ
- Ansibleは、プレイブックの実行中に、`notify`という「通知」を受け取ると、それを内部の実行待ちリスト」に登録する。そして、**全てのタスクが正常に完了した直後** に、そのリストにあるハンドラを一度だけ実行する。
- `roles/common/handlers/main.yml`がハンドラ、`roles/common/`内の各タスクにあるのがnotifyの呼び出し元。
- 特定のタスクで`changed`と判断された時にだけ、実行される。

## ファイルの構成
インストール → 設定ファイルの適用（Template） → サービスの起動設定（Systemd） → コンテンツの配置（Copy）
