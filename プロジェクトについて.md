## 1. プロジェクトの基本方針
プロジェクトの品質を担保する為に、以下の3つの原則を徹底!

- **Single Source of Truth (信頼できる唯一の情報源)**:
	- インフラの状態は全てGit上のコードが正解であるとし、マネジメントコンソールでの手動操作 (ClickOps) を禁止。
- **べき等性の確保**: 
	- 何度実行しても同じ状態になるよう、AnsibleのPlaybookやTerraformの定義を記述。
- **シークレット情報の分割**: 
	- APIトークンやSSH秘密鍵、パスワード類はハードコードせず、環境変数やAnsible Vaultを利用して管理する。

## 2. 構築フェーズ
1. **ローカル開発環境の整備**
		必要な道具を揃える。
		1. **ツールのインストール**: `terraform`及び`ansible`のインストール
		2. **エディタ設定**: NeovimでのHCL (Terraform) 及びYAML (Ansible) のLSP/シンタックスハイライト設定。
		3. **ディレクトリ構造の作成**:
```
vps-iac-pipeline/
├── terraform/          # インフラ（箱）の管理
│   ├── main.tf         # メインの定義
│   ├── variables.tf    # 変数定義
│   ├── outputs.tf      # 実行後に出力する情報（IPアドレス等）
│   ├── providers.tf    # プロバイダー設定
│   └── terraform.tfvars # 【重要】ローカル変数値（Git管理外）
├── ansible/            # OS・ミドルウェアの設定
│   ├── inventory/      # 管理対象サーバーのリスト
│   │   └── hosts.yml
│   ├── playbooks/      # 実行単位（Playbook）
│   │   └── site.yml
│   ├── roles/          # 機能ごとの部品化（再利用性向上）
│   │   ├── common/     # 全サーバー共通設定
│   │   └── nginx/      # Nginx固有設定
│   └── ansible.cfg     # Ansibleの動作設定
├── .gitignore          # Gitに含めないファイルの設定
└── README.md           # プロジェクトの説明書
```
2. **Terraformによるプロビジョニング**
	VPSプロバイダのAPIを利用し、サーバの「外側」を作る。
		1. **Provider定義**: 利用するVPS (さくらのVPS) の認証設定。
		2. **リソース定義**: インスタンス、パケットフィルタ (ファイアウォール)、SSH公開鍵の登録。
		3. **Output設定**: 構築後に割り振られたIPアドレスを自動出力する設定。
		4. **実行検証**: `plan`で内容を確認し、`apply`で実際にサーバが立つことを確認。
3. **作成されたサーバの「中身」を整える**
		1. **Inventory作成**: Terraformで作成されたIPアドレスを管理対象として登録。
		2. **共通設定 (Common)**: ユーザ作成、パッケージアップデート、SSHセキュリティ設定。
		3. **ミドルウェア構築**: Nginxのインストール、systemdによるサービス管理。
		4. **疎通確認**: ブラウザからのVPSのIPにアクセスし、デフォルトページが表示されることを確認。
4. **ポートフォリオとしての仕上げ**
	第三者が「技術力」を判断できる形に整える。
		1. **READMEの執筆**: プロジェクトの目的、構成図、環境構築手順を明文化。
		2. **コードのリファクタリング**: 変数化 (Variables) を適切に行い、再利用性を高める。
		3. **全破壊と再構築テスト**: `terraform destroy`後に再度一から立ち上げ、完全に再現できるか検証。
